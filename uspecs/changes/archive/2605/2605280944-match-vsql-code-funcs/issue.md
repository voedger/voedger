# AIR-3778: Match vsql and code funcs definitions (fiscalcloud fails to start: projector code exists, but missing in vsql)

- **Type**: Task
- **Status**: In Progress
- **Assignee**: d.gribanov@dev.untill.com

## Description

The Jira ticket itself carries no description body beyond the summary; the symptom and cause are recorded here for completeness so that the change request is self-contained.

## Symptom

`Test_FiscalCloud_Vit` (in `airs-bp3`, `packages/air/it/impl_fiscalcloud_test.go`) panics during VVM bootstrap:

```text
panic: runtime error: invalid memory address or nil pointer dereference
...
github.com/voedger/voedger/pkg/vvm.wireVVM.NewSyncActualizerFactoryFactory.func5-range1
    pkg/processors/actualizers/provide.go:43
github.com/voedger/voedger/pkg/istructsmem.(*implIStatelessResources).Projectors
    pkg/istructsmem/resources-types.go:64
github.com/voedger/voedger/pkg/processors/actualizers.NewSyncActualizerFactoryFactory.func5
    pkg/processors/actualizers/provide.go:39
github.com/voedger/voedger/pkg/appparts.newAppPartitionRT
    pkg/appparts/impl_app.go:167
```

Rebuilding the fiscalcloud wasm via `apps/fiscalcloud/build.sh` makes the test pass.

## Cause

`pkg/processors/actualizers/provide.go::NewSyncActualizerFactoryFactory` iterates `statelessResources.Projectors` and looks each one up by `QName` in the deployed app's `IAppDef`:

```go
for _, projector := range statelessResources.Projectors {
    if appdef.Projector(appStructs.AppDef().Type, projector.Name) == nil {
        log.Println()
    }
    if appdef.Projector(appStructs.AppDef().Type, projector.Name).Sync() {
        projectors[projector.Name] = projector
    }
}
```

If a projector is registered in code (`IStatelessResources.AddProjectors`) but is not declared in the vsql baseline that produced `appStructs.AppDef()`, `appdef.Projector(...)` returns `nil`. The `if ... == nil { log.Println() }` branch logs an empty line and falls through, so the next line dereferences `nil` via `.Sync()` and panics.

The fiscalcloud test triggered this because:

- `airs-bp3/go.mod` was bumped to voedger `82dba9f7` (PR #4521, AIR-3704), which adds `sys.ApplyInviteEvents` both to `pkg/sys/sys.vsql` and to `pkg/sys/invite/provide.go::Provide` via `sr.AddProjectors`
- The fiscalcloud image at `vvmdata/apps/untill.fiscalcloud/image/pkg/sys/sys.vsql` was generated by `vpm` from an earlier voedger pin and does not contain the new `ApplyInviteEvents` projector declaration

The runtime side therefore yields a projector that has no counterpart in the deployed `IAppDef`, and the loop panics.

## Scope of the underlying problem

The mismatch surface is wider than the symptom:

- The same risk applies to commands and queries registered via `IStatelessResources.AddCommands` / `AddQueries` whenever they are looked up in `IAppDef` without a `nil` guard
- The risk is not specific to fiscalcloud - any app whose vsql baseline lags the runtime can hit it (and any app whose vsql is ahead of the code can hit the inverse drift)
- The current behaviour - silent drift until a runtime panic in unrelated code - hides the real root cause and forces bisection

## Goal of the fix

Make any divergence between vsql-declared funcs and code-registered stateless funcs detectable at startup, with a clear, composite error that names the offending QNames and the direction of the drift, instead of a `nil` dereference deep in the actualizer factory.
